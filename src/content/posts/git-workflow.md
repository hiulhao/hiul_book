很多人觉得，个人项目直接在 main 分支上提交就行了，搞什么分支管理？

但实际做过几个项目后，我发现分支管理对个人项目同样重要。

## 直接在 main 上提交的问题

### 1. 不敢做大的改动

在 main 分支上做大改动，一旦改到一半发现方向错了，回退很麻烦。你得记得哪些提交是需要回退的，哪些不是。

### 2. 实验性代码污染主线

想试一个新功能，直接写在 main 上。如果不好用想删掉，需要 cherry-pick 或者 reset，很容易出错。

### 3. 没法并行开发

正在写文章 A，突然想到文章 B 的一个点子，想先写下来。如果没有分支，你只能在文章 A 没写完的情况下混入文章 B 的内容。

## 我的个人项目 Git 工作流

### 分支策略

```
main          稳定分支，随时可部署
├── feat/xxx  新功能分支
├── post/xxx  新文章分支
└── fix/xxx   修复分支
```

### 工作流程

1. 从 main 切出分支
2. 在分支上开发和提交
3. 完成后合并回 main
4. 删除分支

```bash
# 写新文章
git checkout -b post/git-workflow
# 写完合并
git checkout main
git merge post/git-workflow
git branch -d post/git-workflow
```

### Commit 规范

即使是个人项目，也建议遵循 Conventional Commits：

```
feat: 添加暗黑模式
fix: 修复标签页分页错误
post: 新增 Git 工作流文章
chore: 更新依赖
```

这样以后看 git log 时一目了然。

## 实际好处

用了分支之后，我发现：

- **心理负担小了**：做大的改动不怕搞坏主线
- **实验更自由了**：试新功能不成功，直接删分支就行
- **上下文更清晰**：一个分支只做一件事，不会被其他事情打断

## 工具辅助

如果你用 VS Code 或类似编辑器，Git 分支操作在 UI 里点两下就行，不用敲命令。

也可以用 `git stash` 临时保存当前工作，切换到别的分支处理紧急问题，处理完再切回来。

```bash
# 临时保存
git stash
# 切到别的分支处理问题
git checkout main
# 处理完切回来
git checkout post/git-workflow
git stash pop
```

## 总结

Git 分支管理不是团队项目的专利。个人项目用分支，成本很低，收益很明显。

> 好习惯的收益是复利的。今天多花 5 秒切分支，未来省 50 分钟回退代码。
